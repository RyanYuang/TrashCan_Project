
## 设计原则

### 1. 分层解耦
- 每层只依赖下层，不依赖上层
- 接口清晰，实现隐藏
- 便于测试和维护

### 2. 配置驱动
- 平台配置与项目配置分离
- 通过宏定义控制功能开关
- 便于不同项目的定制

### 3. 统一管理
- 设备管理器统一管理所有设备
- 应用框架统一管理应用生命周期
- 减少重复代码，提高一致性

### 4. 可扩展性
- 设备可动态注册
- 应用回调可自定义
- 易于添加新功能

## 关键组件说明

### 1. 配置系统

**platform_config.h** - 平台基础配置
- MCU型号和时钟配置
- 调试开关
- 支持的设备列表
- 默认参数

**project_config.h** - 项目配置
- 项目名称和版本
- 项目类型
- 启用的设备
- 覆盖平台默认配置

### 2. 设备管理器

**核心功能**:
- 设备注册与注销
- 设备初始化
- 设备读写控制
- 设备状态管理

**设备描述符**:
```c
DeviceDescriptor_t = {
    .info = {
        .id,        // 设备ID
        .name,      // 设备名称
        .type,      // 设备类型
        .status,    // 设备状态
        .handle,    // 设备句柄
        .config     // 设备配置
    },
    .ops = {
        .init,      // 初始化函数
        .deinit,    // 反初始化函数
        .read,      // 读取函数
        .write,     // 写入函数
        .control    // 控制函数
    }
}
```

### 3. 应用框架

**生命周期**:
1. **UNINIT** → 未初始化
2. **INIT** → 初始化中（on_init）
3. **RUNNING** → 运行中（on_loop循环执行）
4. **PAUSED** → 暂停（on_pause）
5. **ERROR** → 错误（on_error）
6. **SHUTDOWN** → 关闭（on_stop）

**回调函数**:
- `on_init()` - 初始化应用资源
- `on_start()` - 应用启动
- `on_loop()` - 主循环逻辑
- `on_pause()` - 暂停应用
- `on_resume()` - 恢复应用
- `on_stop()` - 清理资源
- `on_error()` - 错误处理

## 数据流

```
传感器 → 设备驱动 → 设备管理器 → 应用框架 → 项目应用
  ↓                                            ↓
ADC/I2C                                    数据处理
  ↓                                            ↓
原始数据                                    业务逻辑
  ↓                                            ↓
                                            显示/上报
                                               ↓
                                           执行器控制
                                               ↓
                                         设备管理器
                                               ↓
                                           设备驱动
                                               ↓
                                        电机/LED/蜂鸣器
```

## 使用流程

### 创建新项目

1. **复制模板**
   ```bash
   cp -r Projects/Template Projects/NewProject
   ```

2. **修改配置**
   - 修改项目名称、版本
   - 选择需要的设备
   - 配置项目参数

3. **实现回调**
   - on_init: 初始化设备
   - on_loop: 实现主逻辑

4. **调用应用**
   ```c
   void User_Main(void)
   {
       NewProject_App_Main();
   }
   ```

### 添加新设备

1. **编写设备驱动** (Device/)
2. **定义设备ID** (device_manager.h)
3. **实现设备操作** (init/read/write/control)
4. **注册设备** (项目初始化中)
5. **使用设备** (通过设备管理器)

## 优势

### 对比原架构

| 特性 | 原架构 | 新架构（平台化） |
|------|--------|-----------------|
| 代码复用 | 低（每个项目重写） | 高（基于平台和模板） |
| 开发效率 | 慢（从零开始） | 快（使用模板） |
| 设备管理 | 分散（各处直接调用） | 统一（设备管理器） |
| 配置管理 | 混乱（硬编码） | 清晰（配置文件） |
| 可维护性 | 差（代码耦合） | 好（分层解耦） |
| 可扩展性 | 差（修改困难） | 好（易于扩展） |
| 代码一致性 | 低（风格各异） | 高（统一框架） |

### 平台化带来的好处

1. **快速开发** - 使用模板，几分钟创建新项目
2. **代码复用** - 平台代码在所有项目中复用
3. **统一管理** - 设备和应用统一管理
4. **易于维护** - 清晰的架构和分层
5. **降低门槛** - 新项目只需关注业务逻辑
6. **提高质量** - 统一的框架保证代码质量
   
## 最佳实践

1. **项目代码只关注业务逻辑**
   - 设备初始化放在 InitDevices()
   - 业务逻辑放在 OnLoop()
   - 不直接操作硬件寄存器

2. **使用设备管理器**
   - 统一通过设备管理器访问设备
   - 不直接调用底层驱动

3. **配置化开发**
   - 功能通过配置文件开 b关
   - 参数通过配置文件定义

4. **遵循生命周期**
   - 初始化在 on_init
   - 清理在 on_stop
   - 错误处理在 on_error

5. **代码组织**
   - 项目代码放在 Projects/
   - 设备驱动放在 Device/
   - 底层驱动放在 Driver/
